# 前端浏览器测试规范

- 每次新增或修改页面、组件后，**必须在浏览器中进行实际测试**，不能只做编译检查。
- 浏览器测试是交付前的必要步骤，确保 UI 渲染正确、交互逻辑正常。

## 测试前置条件

1. **启动开发服务器**：确保 `blog-home-nuxt` 开发服务器已运行（`npm run dev`）。
2. **确保后端服务可用**：如果页面涉及 API 调用，`blog-server` 也需启动。
3. **清除缓存**：如果涉及接口数据变更，需确认数据已正确写入数据库。

## 浏览器测试方式

### 方式一：使用 browser-use MCP 工具（推荐）

- 通过 `browser-use` MCP 工具自动化浏览器操作。
- 典型流程：导航到目标页面 → 截图验证 UI → 执行交互操作 → 截图确认结果。
- 重点检查：
  - 页面是否正确渲染（无白屏、无报错）
  - 组件布局是否正常（响应式、对齐、间距）
  - 交互是否生效（按钮点击、表单提交、下拉菜单展开）
  - API 数据是否正确展示（列表加载、状态显示）

### 方式二：手动浏览器测试

- 打开浏览器访问目标页面 URL。
- 检查控制台是否有报错（红色 Error）。
- 手动操作验证功能是否正常。

## 测试覆盖要求

- **新页面**：访问页面 → 验证整体布局 → 测试核心功能。
- **新组件**：在页面中确认组件正确渲染 → 验证组件交互。
- **导航入口**：确认新页面的入口链接可点击且跳转正确。
- **登录态**：分别测试已登录和未登录状态下的页面表现。
- **响应式**：检查移动端和桌面端的布局是否合理。

## 测试失败处理

- 如果浏览器测试发现白屏、JS 报错或 UI 异常，必须修复后重新测试。
- 记录控制台错误信息，定位问题根因。
- 修复后需要再次浏览器测试确认问题已解决。
